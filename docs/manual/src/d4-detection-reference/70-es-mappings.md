# 13. Elasticsearch mappings reference

ChainSentinel uses two strictly-mapped indices. The full field-by-field
tables are generated by `scripts/generate_catalog_tables.py` parsing
`chainsentinel/es/mappings/*.json`. Regenerate before building this
document if the mappings change.

## 13.3 Shared fields (always present)

Every document in either index has:

| Field | Type | Notes |
|-------|------|-------|
| `investigation_id` | `keyword` | UUID prefixed by run timestamp |
| `chain_id` | `long` | 31337 for Anvil, 1 for mainnet, 11155111 for Sepolia |
| `@timestamp` | `date` | ISO 8601 UTC |
| `block_number` | `long` | The block the evidence came from |
| `block_datetime` | `date` | ISO 8601 UTC |
| `tx_hash` | `keyword` | Lowercased `0x…` |
| `layer` (in `forensics`) | `keyword` | `decoded`/`derived`/`signal`/`alert`/`attacker`/`case` |
| `doc_type` (in `forensics-raw`) | `keyword` | `transaction`/`log`/`trace` |

## 13.4 Index lifecycle

Indices are created by `chainsentinel/es/setup.py` on the first
`POST /analyze` request and again by `start.sh` when ES boots. Existing
data is preserved unless the operator explicitly deletes the indices.

For very long-running investigations or shared deployments, consider
rolling indices weekly:

```
PUT _ilm/policy/forensics-policy
{
  "policy": {
    "phases": {
      "hot":    { "actions": { "rollover": { "max_age": "7d", "max_size": "20gb" }}},
      "delete": { "min_age": "180d", "actions": { "delete": {} }}
    }
  }
}
```

This is **not** enabled by default — analysts typically keep all data.
